# CC vs Codex 对比测试 — 设计文档

**日期**：2026-04-28
**状态**：设计完成，待启动
**载体任务**：PI / 晨报升级（端到端，brainstorm + plan + 实现）

---

## 目的

在做真活的同时，回答两个问题：

1. **能力对比**（A）— 同样原始需求下，cc 和 codex 谁的端到端工作成果更好？
2. **效率对比**（B）— 谁更快完成（壁钟时间）？

非目标：不追求"评估底层模型能力"——两边的 harness（CLAUDE.md / MEMORY.md / AGENTS.md / 工具链）已配置为完全对等，对比的就是"同条件下 cc 工作流 vs codex 工作流"的综合产出。

---

## 核心设计：端到端独立执行

不切子任务，**整个 PI/晨报升级就是对比内容**。两边各自从零开始：自己 brainstorm 需求、自己写 plan、自己实现代码。

| 阶段 | 两边各自做什么 |
|------|---------------|
| 1. 接收原始问题 | Boss 在两边新对话里发同一个"原始需求"prompt |
| 2. brainstorm | 各自挖需求、提问澄清，最后理解到同一组需求点上 |
| 3. plan | 各自写 plan 文档（架构图、流程图、checklist 等） |
| 4. 实现 | 各自在自己的 worktree 里写代码 |
| 5. 提交 | 完成后 commit 到各自 worktree 分支 |

预期总耗时：4-8 小时/边（端到端）。

---

## 答疑规则（关键）

brainstorm 和 plan 阶段两边都会问 Boss 设计选择题（"用 hourly 还是 daily？"、"要不要加 X 维度？"）。

**铁律：两边给完全一致的回答**。哪怕两边问法不同，Boss 也尽量把答案映射到同一个语义。

**操作规范**：
- Boss 在自己脑子里维护一份"已答清单"——每次答疑时记下"问题 → 答复"
- 当另一边问类似问题时，查清单给一致答复
- 如果某一边没问到、另一边问到了，**不要主动告诉前者**（让两边的"主动追问"能力体现在对比里）
- 实施过程中可能 Boss 会发现某些回答需要修正——修正后两边都同步告知

**事实性追问**（"这个文件叫什么"、"有没有现成的 helper"）正常回答，无需平衡。

---

## Worktree 与匿名化

- 实施前**抛硬币决定**：A 分配给谁、B 分配给谁。Boss 心里记着对应关系，但不告诉任何 agent。
- 两边各在自己 worktree 里独立工作（互不感知对方的存在）
- worktree 命名：`finance-comparison-A` / `finance-comparison-B`（或类似中性名）
- 完成后 commit，diff 用 worktree 名字标识

---

## 评分机制：三方评定

**两份 worktree 完成后**，三方各自评分：

| 评分方 | 看到的内容 | 是否盲 |
|--------|------------|--------|
| cc | A 和 B 两份 diff（不知道哪份是自己写的） | **盲** |
| codex | A 和 B 两份 diff（不知道哪份是自己写的） | **盲** |
| Boss | A 和 B 两份 diff | **明**（操作者无法真正盲，承认这一点） |

**评分维度**（每项 1-5 分 + 简短文字理由）：
1. 需求理解（brainstorm 是否抓到核心、plan 是否对齐需求）
2. 设计质量（plan 的架构合理性、决策是否有依据）
3. 代码质量（可读、简洁、无过度工程化）
4. 项目惯例契合（是否复用已有 helper、风格符合 CLAUDE.md）
5. 完成度（功能是否真的能跑、有无烂尾）

**总评**：1-10 分 + 一段亮点/槽点笔记。

**要观察的元信号**：
- cc 和 codex 是否系统性偏向自己的方案（自我偏见）
- 三方分歧最大的维度（暴露评估视角差异）
- Boss 的判断更接近哪个 agent（间接验证 agent 评估能力）

## 执行流程

| 步骤 | 动作 |
|------|------|
| 1 | Boss 写好"原始需求"prompt（PI/晨报升级的需求陈述，**不暴露任何实现思路**） |
| 2 | Boss 抛硬币决定 A/B 分配，记下对应关系 |
| 3 | 在 cc 和 codex 各开新对话，同时投递同一份 prompt |
| 4 | 两边各自 brainstorm，Boss 按"一致答复"铁律答疑 |
| 5 | 两边各自 commit plan，Boss 不审核（让 plan 也算对比内容） |
| 6 | 两边各自创建 worktree，开始实现 |
| 7 | 实现期间 Boss 继续按"一致答复"铁律答疑 |
| 8 | 两边都完成（或到时间上限），各自 commit |
| 9 | 把两份 diff 匿名化为"方案 X / 方案 Y"，分别喂给 cc 和 codex 各开新对话做盲评 |
| 10 | Boss 自己也明评一份 |
| 11 | 揭晓 + 三方评分对照 + 写复盘到 `docs/postmortems/` |

## YAGNI 检查

不做的事：
- 不切子任务（Boss 选择端到端对比）
- 不做 acceptance 硬验证（任务规模没法用一行 CLI 验证）
- 不做"自动化打分脚本"
- 不做多轮对比
- 不做统计显著性检验

---

## 风险

| 风险 | 缓解 |
|------|------|
| brainstorm 阶段 Boss 答复不一致，污染对比 | "一致答复"铁律 + 维护答疑清单；事后复盘时检查 qa_log 一致性 |
| 一边或两边在 8 小时内没完成 | 设上限 8 小时，到点强制提交"现状快照"，记"未完成范围" |
| Boss 同时左右开弓答疑会累 | 接受这个成本，否则没法保证"同时进行"和"答疑一致性" |
| Boss 无法真正盲评 | 承认这点，把 Boss 评分权重看成"明评 reference"，重点看 cc/codex 互评的盲信号 |
| 两个 agent 都打压对方/抬高对方 | 三方对照本身就是为了暴露这一点 |
| 原始 prompt 暴露实现思路 → 抹平差异 | Boss 写 prompt 时只描述"要解决什么问题"，不写"应该怎么做"|
